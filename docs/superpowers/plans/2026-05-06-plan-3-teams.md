# Plan 3 — Teams (Outline)

> Tight outline. Per-task detail expanded during execution.
> Source of truth: `docs/superpowers/specs/2026-05-04-trivix-foundation-design.md`, sections 3 (team flows) + 4 (data model) + 5 (admin actions on teams) + 6 (routing).

## Goal

Ship team creation, invite-code joining, captain-approved membership, captain transfer, and team browsing for members. End state: a verified user can create a team, share an invite code, approve/deny join requests, transfer captaincy, regenerate the code, and leave or disband. Players can join via invite code and see real-time updates while pending.

## Out of scope (deferred)

- Admin teams list / admin team delete UI — Plan 4
- Public team browse list — spec note line 27 says "invite-code only," so no browse page
- Team chat / activity feed
- Per-team theming or avatars (just initials + name for slice 1)

## Prerequisites (already done in Plan 2)

- `users/{uid}` carries `teamId`, `teamHistory`
- Auth, email verification gate, `(app)` layout guard, middleware
- UI primitives, motion tier, design system

## Task list

Numbering picks up from Plan 2 (which ended at Task 69).

### Foundation

- **70. Zod schemas + types** — `lib/validation/schemas.ts` adds `createTeamSchema` (name), `joinTeamSchema` ({ inviteCode }), `transferCaptainSchema` ({ uid }). `types/firestore.ts` adds `TeamDoc`, `JoinRequestDoc`, `SerializedTeam`.
- **71. Invite-code generator** — `lib/teams/invite-code.ts`: 6-char A-Z + 0-9, ambiguous chars excluded (no `0/O`, `1/I/L`). Pure function with seedable RNG for tests.
- **72. Firestore rules v3** — open `teams/{teamId}` for member reads, captain writes via Admin SDK; `joinRequests/{uid}` allows owner-create + captain-read; everything else server-only. Update rules tests with 8–10 new cases.
- **73. Composite index** — `teams` by `inviteCode` (single-field auto-indexed, but membership query needs composite: `users.teamId == X` + `users.displayName asc`). Add to `firestore.indexes.json` once we know the exact dashboard query.

### Server routes (all behind session-cookie auth)

- **74. `POST /api/teams`** — create team. Verifies session, generates invite code with retry on collision (read sentinel `teams` by `inviteCode`), atomic transaction creates `teams/{teamId}` + updates caller's `users/{uid}.teamId` + appends to `teamHistory`. Errors if user already on a team.
- **75. `POST /api/teams/join`** — body `{ inviteCode }`. Looks up team by code; if not found → 404. If user already on a team → 409. Creates `joinRequests/{uid}` subdoc with payload.
- **76. `GET /api/teams/[id]`** — returns serialized team (members resolved to display names + avatars). 403 if caller is not a member or captain.
- **77. `POST /api/teams/[id]/leave`** — leave team. Branches on `memberUids.length`: sole member disbands the team; otherwise removes from `memberUids`, clears `users/{uid}.teamId`, sets `captainUid: null` if the leaver was captain (per D1).
- **77b. `POST /api/teams/[id]/claim-captain`** — member-only. 409 if `captainUid !== null`. Atomic update sets `captainUid` to caller.
- **78. `POST /api/teams/[id]/transfer-captain`** — body `{ uid }`. Captain-only. Target must be a current member. Atomic captain swap.
- **79. `POST /api/teams/[id]/regenerate-code`** — captain-only. New 6-char code, retry on collision. Atomic update.
- **80. `GET /api/teams/[id]/requests`** — captain-only. Lists pending `joinRequests` with requester display names.
- **81. `POST /api/teams/[id]/requests/[uid]`** — captain-only. Body `{ action: 'approve' | 'deny' }`. Approve = transaction: add uid to `memberUids`, set `users/{uid}.teamId`, append `teamHistory`, delete the request doc. Deny = just delete.
- **82. `DELETE /api/teams/[id]`** — captain-only disband. Transaction: clear `users/*.teamId` for all members, delete request subcollection, delete team doc.

### Pages

- **83. `(app)/team/page.tsx`** — current-team view if `user.teamId`, otherwise no-team CTA panel. Server component, fetches team doc.
- **84. `(app)/team/create/page.tsx`** — name field + create button. On success, redirect to `/team`.
- **85. `(app)/team/join/page.tsx`** — invite-code entry. On success, redirect to `/team/pending`.
- **86. `(app)/team/pending/page.tsx`** — "request submitted, waiting for captain" with `onSnapshot` real-time wait. On approval, push to `/team`.
- **87. `(app)/team/settings/page.tsx`** — captain-only. Shows invite code, member list, pending requests, regen code button, transfer dropdown, disband team button.

### Tests

- **88. Unit: invite-code generator** — randomness, length, character set, collision-resistance smoke (10k generated, < N collisions).
- **89. Integration: rules** — captain reads requests, non-captain can't, member reads team, non-member can't. Owner can create own joinRequest, can't create someone else's.
- **90. Integration: routes** — happy paths + error cases for create, join, approve/deny, transfer, leave, regen, disband. Includes sole-member leave deletes team.
- **91. E2E: team smoke** — Playwright spec verifies `/team`, `/team/create`, `/team/join`, `/team/settings` render with mock authed cookie.

### Final

- **92. Verification pass** — format, lint, typecheck, unit, integration vs emulator, build, E2E. Tag `plan-3-complete`.

## Acceptance criteria

1. Verified user can create a team; gets a 6-char invite code on success.
2. Verified user can submit a join request via invite code; sees pending state.
3. Captain can view, approve, and deny join requests; approved members appear in member list.
4. Captain can transfer the captain role to any current member.
5. Captain can regenerate the invite code; old code stops working immediately.
6. Captain can leave (transfers if other members exist; disbands if alone).
7. Captain can disband team; all members' `teamId` clears.
8. Non-members cannot read team data via direct Firestore client reads.
9. Captain-only routes return 403 to non-captains; member-only routes return 403 to non-members.
10. All test layers green; new emulator-backed E2E for team smoke.

## Decisions (resolved 2026-05-06)

- **D1.** Captain leave with other members present → `captainUid` goes null; team continues. Any remaining member can call a `POST /api/teams/[id]/claim-captain` route to take over. Sole-member captain leave → team is disbanded.
- **D2.** Invite code: 6 chars from a 31-char alphabet (digits + uppercase, excluding `0/O/1/I/L`). 31^6 ≈ 887M combinations.
- **D3.** No rate-limiting on team creation in slice 1. Revisit if abuse appears.
- **D4.** `/team/pending` uses client-side `onSnapshot` on the caller's own `users/{uid}` doc; when `teamId` flips non-null, push to `/team`.

> Task 77 (leave) updated: branch on `memberUids.length` — sole member disbands, otherwise null out captain. Add Task 77b: `POST /api/teams/[id]/claim-captain` (member-only, requires `captainUid === null`).
