# Plan 4 — Admin Portal (Outline)

> Tight outline. Per-task detail expanded during execution.
> Source of truth: `docs/superpowers/specs/2026-05-04-trivix-foundation-design.md`, sections 3, 4, 5, 6, 9.

## Goal

Ship the lean admin portal: pending host-application approval, all-users list with delete + revoke-pro, all-teams list with delete. End state: a founder with `isAdmin: true` can land at `/admin`, work the host queue, manage users, and disband teams. Non-admins get redirected away from `/admin/*`.

## Out of scope (deferred)

- Audit logs (spec line 31 calls out)
- Role-tiered admin permissions (only one admin level in slice 1)
- Impersonation
- Admin-promotion UI (founder bootstraps via Firestore console; no in-app way to add admins per spec line 539)
- Pagination/search beyond simple client-side filter

## Prerequisites (already done)

- `users/{uid}.isAdmin` field exists, defaults `false`; founder set manually via console (per spec)
- `hostApplications` collection populated by signup wizard (Plan 2 Task 42)
- Teams + memberships shipped in Plan 3
- Middleware matches `/admin/:path*` and gates on session cookie
- `Badge` primitive supports `host`, `pending`, `pro` tones

## Task list

Numbering picks up from Plan 3 (which ended at Task 92).

### Foundation

- **93. Zod schemas** — `lib/validation/schemas.ts` adds `hostApplicationActionSchema` (`{ action: "approve" | "deny" }`) and `userActionSchema` (`{ action: "revoke-pro" | "delete" }`). Maybe more as routes land.
- **94. `(admin)/layout.tsx` server guard** — verifies session + `users/{uid}.isAdmin === true`, otherwise `redirect("/dashboard")`. Provides admin chrome (calmer styling, sidebar).
- **95. Firestore rules update** — add admin override: `function isAdmin() = exists(/users/$(uid)) && get(/users/$(uid)).data.isAdmin == true`. Allow admin to read all of `users`, `teams`, `hostApplications`. Writes still server-only. Add 6–8 rules tests.

### Admin server routes

- **96. `GET /api/admin/host-applications`** — list pending applications (admin only). Returns serialized list with requester display name + email + reason + appliedAt.
- **97. `POST /api/admin/host-applications/[uid]`** — body `{ action }`. Approve sets `users/{uid}.hostStatus = "approved"`, `hostApplications/{uid}.status = "approved"` + `decidedAt` + `decidedBy`. Deny sets to `"denied"`. Atomic transaction.
- **98. `GET /api/admin/users`** — list all users (admin only). Returns slim payload: uid, email, displayName, role, hostStatus, isAdmin, teamId, createdAt.
- **99. `POST /api/admin/users/[uid]`** — body `{ action }`. `revoke-pro` sets `hostStatus = "none"`, `role = "player"`. `delete` runs full cascade: leave team (using existing leave logic if member), delete `users/{uid}`, delete `displayNames/{key}`, delete `hostApplications/{uid}` if any, delete Firebase Auth user.
- **100. `GET /api/admin/teams`** — list all teams (admin only). Slim payload with member count + captain display name.
- **101. `DELETE /api/admin/teams/[id]`** — admin disband. Reuses Task 82's logic but bypasses captain check. Cascades clear of all `users.teamId` and join requests.

### Pages

- **102. `(admin)/admin/page.tsx`** — overview. Counts: pending host apps, total users, total teams. Quick links to each section.
- **103. `(admin)/admin/host-applications/page.tsx`** — server-rendered list with approve/deny client buttons.
- **104. `(admin)/admin/users/page.tsx`** — searchable user list (client-side filter). Per-row dropdown: revoke-pro, delete. Confirm modal for delete (typed confirmation per spec §7).
- **105. `(admin)/admin/teams/page.tsx`** — team list with delete action + captain transfer notice.

### Components

## Acceptance criteria

1. Visiting `/admin/*` as a non-admin redirects to `/dashboard`.
2. Admin lands at `/admin` and sees overview counts.
3. Admin can approve a pending host application; user's `hostStatus` flips to `approved` and they see the host badge.
4. Admin can deny a host application; status flips to `denied`.
5. Admin can revoke a user's pro status; user becomes a regular player.
6. Admin can delete a user; their `users` doc, displayName sentinel, host application, and Firebase Auth account are all removed; if they were on a team, they're cleanly removed (sole-member teams disband).
7. Admin can delete a team; all members' `teamId` cleared, join requests gone, team doc gone.
8. Firestore rules allow admin to read `users`, `teams`, `hostApplications`; non-admins still denied.
9. All test layers green; CI green.

## Decisions to lock in before execution

- **D1.** When admin deletes a user who is captain of a team with other members, do we transfer captaincy automatically (to oldest member by `joinedAt`?), set `captainUid = null` (Plan 3 D1 pattern), or block the delete? *Recommend: set `captainUid = null`. Same pattern as captain leaving — remaining members can claim.*
- **D2.** "Delete" a user — also delete from Firebase Auth, or just Firestore docs? *Recommend: both. Otherwise they can sign back in and become an orphan.*
- **D3.** Admin actions audit log — punt to a later slice (per spec line 31), or stash a minimal `auditLogs` collection now? *Recommend: punt. Spec explicitly calls it out as deferred.*
- **D4.** "Revoke pro" terminology — spec uses both "host" and "pro." Confirm we mean `hostStatus = "none"` + `role = "player"` rollback. *Recommend: yes; one button labeled "Revoke host."*
- **D5.** Admin layout chrome — separate sidebar nav, or a tab strip on the overview page? *Recommend: tab strip + sticky header. Lean enough for slice 1.*
